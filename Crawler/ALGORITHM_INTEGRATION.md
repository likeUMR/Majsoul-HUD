# Crawler 接入 `mahjong-cpp` 说明

本文档基于：

- `Algorithm/mahjong-cpp-master/INTERFACE_SPEC.md`
- 当前 `Crawler` 已实现的状态采集逻辑

目标是回答两个问题：

1. `Crawler` 目前能识别并传入算法的哪些字段
2. 这个算法能否满足“摸牌后推荐出牌”和“吃碰杠机会时给建议”的需求

## 1. 结论先说

结论分两部分：

- `摸牌后推荐出牌`：可以做，而且契合度较高，是最适合优先落地的部分。
- `吃/碰/杠机会推荐`：可以做，但不是“直接传一个操作机会，算法直接回答要不要吃碰杠”。需要 `Crawler` 先枚举每个可选操作，把操作后的局面模拟出来，再分别调用算法比较结果。

也就是说：

- 这个算法天然擅长 `当前局面 -> 推荐切哪张`
- 这个算法不天然擅长 `当前有吃碰杠机会 -> 直接输出要不要吃碰杠`

但只要 `Crawler` 负责做一层“操作分支模拟”，你的需求仍然是可实现的。

## 2. 算法接口需要什么

根据接口文档，算法核心请求字段如下：

| 字段 | 是否需要 | 含义 |
| --- | --- | --- |
| `enable_reddora` | 必填 | 是否启用赤宝牌 |
| `enable_uradora` | 必填 | 是否启用里宝牌期望值 |
| `enable_shanten_down` | 必填 | 是否允许向听倒退搜索 |
| `enable_tegawari` | 必填 | 是否允许手变搜索 |
| `enable_riichi` | 必填 | 是否考虑立直 |
| `round_wind` | 必填 | 场风 |
| `seat_wind` | 必填 | 自风 |
| `dora_indicators` | 必填 | 宝牌指示牌 |
| `hand` | 必填 | 自家当前手牌 |
| `melds` | 必填 | 自家当前副露 |
| `version` | 必填 | 算法版本字符串 |
| `wall` | 可选但强烈建议 | 剩余未知牌计数 |

## 3. 当前 Crawler 能提供什么

### 3.1 已经能稳定拿到的字段

当前 `Crawler` 已经可以从雀魂协议中识别并维护这些信息：

| 算法字段 | Crawler 现状 | 说明 |
| --- | --- | --- |
| `hand` | 可直接提供 | 当前已维护自家手牌，包含开局手牌、摸牌后手牌、吃碰杠后的手牌变化 |
| `melds` | 可直接提供 | 当前已维护自家副露 |
| `dora_indicators` | 可直接提供 | 新局和后续开杠翻宝牌时都能拿到 |
| `round_wind` | 可推导 | 可由 `ActionNewRound.chang` 推导 |
| `seat_wind` | 可推导 | 可由 `self_seat` 和 `ju` 推导 |
| `wall` | 可构造近似值 | 当前已维护已现牌统计，可据此反推剩余枚数 |

### 3.2 需要静态配置或额外补充的字段

这些字段不是牌局状态本身，而是“算法开关”或“运行配置”，应由接入层配置：

| 算法字段 | 当前来源 | 建议 |
| --- | --- | --- |
| `enable_shanten_down` | 静态配置 | 先固定为 `true` |
| `enable_tegawari` | 静态配置 | 先固定为 `true` |
| `enable_riichi` | 静态配置 | 可传 `true`，且文档说明当前实现里本来也会被强制开启 |
| `version` | 静态配置 | 必须与算法服务实际编译版本一致 |
| `ip` | 可不传 | 可选 |

### 3.3 当前还不够严谨的部分

这些字段理论上能传，但当前 `Crawler` 还没有完整规则层，因此只能先“默认”或“近似”：

| 算法字段 | 当前状态 | 风险 |
| --- | --- | --- |
| `enable_reddora` | 可默认 `true` | 对大多数四麻日麻模式没问题，但若房规禁赤，需要补规则识别 |
| `enable_uradora` | 可默认 `true` | 对大多数立直麻将模式通常成立，但若房规禁里宝，需要补规则识别 |
| `wall` | 可近似构造 | 若只按已现牌统计，仍比算法自动估算更真实，但不代表完整“未知牌山” |

## 4. 字段如何从 Crawler 映射到算法

## 4.1 牌编码转换

`Crawler` 当前内部主要使用雀魂风格字符串牌表示，例如：

- `1m..9m`
- `1p..9p`
- `1s..9s`
- `1z..7z`
- `0m/0p/0s` 表示赤五

而算法要求的是整数编码：

| 字符串牌 | 算法编码 |
| --- | --- |
| `1m..9m` | `0..8` |
| `1p..9p` | `9..17` |
| `1s..9s` | `18..26` |
| `1z..7z` | `27..33` |
| `0m` | `34` |
| `0p` | `35` |
| `0s` | `36` |

所以接入时需要先实现一个稳定的 `tile_str -> tile_int` 转换函数。

## 4.2 副露映射

算法的 `melds` 结构是：

```json
{
  "type": 0,
  "tiles": [0, 0, 0]
}
```

当前 `Crawler` 内部副露类型与算法可做如下映射：

| Crawler 副露 | 算法 `type` |
| --- | --- |
| 碰 | `0` |
| 吃 | `1` |
| 暗杠 | `2` |
| 明杠 | `3` |
| 加杠 | `4` |

注意：

- `Crawler` 内部副露牌目前是字符串数组，需要整体转成整数数组
- 吃碰杠组合的合法性应由 `Crawler` 保证，算法不会替你纠错

## 4.3 场风和自风

算法要求：

- `round_wind` 取 `27/28/29/30`
- `seat_wind` 取 `27/28/29/30`

而 `Crawler` 当前能拿到：

- `chang`：场风序号
- `ju`：本局庄家位序
- `self_seat`：自家座位

建议映射：

- `round_wind = 27 + chang`
- `seat_wind = 27 + ((self_seat - ju + 4) % 4)`

说明：

- `chang=0/1/2/3` 对应东/南/西/北场
- `ju` 可理解为当前庄家所在 seat
- `self_seat` 确认后即可推出自风

## 4.4 `wall` 如何构造

算法文档明确说明：如果不传 `wall`，它只会根据以下信息做默认估算：

- 手牌
- 副露
- 宝牌指示牌

不会自动扣除：

- 所有人弃牌
- 其他公开信息

而当前 `Crawler` 已经维护了较完整的“已现牌”统计，来源包括：

- 自家手牌
- 自家副露
- 其他玩家打出的牌
- 其他玩家副露中公开可见的牌
- 宝牌指示牌

因此，`Crawler` 很适合自己构造 `wall`。

推荐做法：

1. 先为 37 种编码建立初始剩余数
2. 普通牌默认 `4`
3. 赤牌 `0m/0p/0s` 默认 `1`
4. 用当前已现牌统计逐项扣减
5. 得到最终的 `wall[37]`

注意事项：

- 普通 `5m/5p/5s` 与赤 `0m/0p/0s` 要分开扣减
- 但对 `mahjong-cpp` 来说，赤五同时也会占用对应普通五的总张数
- 所以若 `0m/0p/0s` 已现，构造 `wall` 时除了扣赤牌位，也要同步扣 `5m/5p/5s`
- `wall` 不能出现负数
- 若某牌统计超过理论上限，算法会直接报错

## 5. Crawler 什么时候调用算法

## 5.1 摸牌后推荐出牌

这是最自然的接入点。

建议触发时机：

- 自家 `ActionDealTile` 后
- 或者任何导致“当前轮到自家出牌”的状态更新后

此时 `Crawler` 可整理出：

- `hand`：14 张或副露手对应的当前闭手牌
- `melds`
- `round_wind`
- `seat_wind`
- `dora_indicators`
- `wall`

然后调用算法，读取：

- `response.shanten`
- `response.stats[]`

在 HUD 上展示：

- 推荐切牌
- 切后向听
- 有效牌列表
- 有效牌总数
- 某一巡的听牌率 / 和了率 / 期待值

这个场景下，算法是非常匹配的。

补充注意：

- 对于无副露手，轮到出牌时通常就是 `hand=14`
- 但对于已经吃碰后的副露手，轮到出牌时闭手张数会变成 `11/8/5`
- 接口文档对“13 张与 14 张”的描述更像是无副露场景下的说明
- 从接口存在 `melds` 与 `num_tiles` 的设计来看，它理论上应当支持副露手
- 但在真正接入前，最好用一手“已吃/已碰”的实战样例请求一次算法服务，确认它仍会返回可用于推荐切牌的 `stats`

## 5.2 吃碰杠机会时给建议

这个需求能做，但需要明确是“间接支持”。

原因是算法输入里没有这种字段：

- “当前有一个碰机会，请直接回答要不要碰”
- “当前有两种吃法，请直接帮我选一种”

它只接受“一个已经确定的局面”，然后告诉你这个局面下怎么打更优。

因此正确做法应是：

1. `Crawler` 从 `operation.operation_list` 中识别出所有可选操作
2. 对每个操作组合，模拟“执行该操作后的自家手牌与副露”
3. 对每个模拟分支分别调用算法
4. 比较各分支结果，给出建议

例如遇到碰牌机会时，可以比较：

- `不碰`：保持当前闭手，等待后续
- `碰后打牌`：构造新 `melds` 和新 `hand`，再跑算法

遇到吃牌机会时，可以比较：

- `不吃`
- `吃法 A + 切牌推荐`
- `吃法 B + 切牌推荐`

遇到杠牌机会时，可以比较：

- `不杠`
- `加杠`
- `暗杠`
- `大明杠`

再依据返回的：

- `shanten`
- `necessary_tiles`
- `tenpai_prob`
- `win_prob`
- `exp_score`

做排序。

## 6. 这个算法能否满足你的需求

## 6.1 能很好满足的部分

### A. 摸牌后推荐切牌

这是它最强的主场。

原因：

- 接口本来就是围绕“当前手牌怎么打”设计的
- 会返回每个候选舍牌的独立结果
- 能提供有效牌、向听、听牌率、和了率、期待值

所以：

- `摸牌后推荐出牌`：可以认为完全满足

### B. 副露后的打牌建议

如果你已经决定吃/碰/杠，算法也很适合给“副露后打什么”的建议。

所以：

- `吃了以后打哪张`
- `碰了以后打哪张`
- `加杠后后续形状如何`

这些都属于它擅长范围。

## 6.2 能满足，但需要 Crawler 多做一层工作的部分

### A. 吃碰杠是否执行

这部分不是算法直接输出，而是 `Crawler` 需要做“分支评估器”：

- 分支 1：不操作
- 分支 2：执行操作 A
- 分支 3：执行操作 B

然后比较未来结果。

所以：

- `要不要吃`
- `要不要碰`
- `两种吃法哪种更优`
- `要不要杠`

这些都可以做，但工作量在 `Crawler` 侧，不在算法接口本身。

## 6.3 暂时不能指望它直接解决的部分

### A. 防守 / 放铳风险评估

从接口文档看，它主要是：

- 牌效
- 听牌率
- 和了率
- 期待值

并没有明确提供：

- 对手危险度
- 振听风险分析
- 现物/筋/壁等防守判断
- “这张牌更安全”这类防守模型

所以如果你以后想做“进攻推荐 + 防守推荐”双系统，还需要额外逻辑。

### B. 直接输出“是否立直/是否鸣牌”的最终决策文案

它能给你计算依据，但不直接给完整行为策略标签。

例如：

- 它不会直接返回“建议碰”
- 也不会直接返回“建议默听，不建议立直”

这些标签仍需要 `Crawler/HUD` 自己根据返回值二次加工。

## 7. 建议的接入顺序

为了最快看到效果，建议按下面顺序做：

1. 先接“摸牌后推荐切牌”
2. 只传最小必要字段：`hand`、`melds`、`round_wind`、`seat_wind`、`dora_indicators`、`version`
3. 跑通后再补 `wall`
4. 最后再做“吃碰杠分支模拟”

这样做的原因是：

- 第 1 步最容易验证算法是否正常工作
- 第 2 步就已经能产出可见的 HUD 推荐
- `wall` 会明显提升质量，但不是链路打通的前置条件
- 鸣牌推荐是第二阶段功能，更适合等基础出牌推荐稳定后再做

## 8. 你这个项目里的实际建议

基于你现在的目标，我的建议是：

### 第一阶段

先实现：

- 自家摸牌后，HUD 显示前 3 个推荐切牌
- 每个候选显示：
  - 切哪张
  - 向听
  - 有效牌总数
  - 某一巡期待值

### 第二阶段

再实现：

- 出现吃/碰/杠机会时
- `Crawler` 读取 `operation_list`
- 自动枚举每个可行操作分支
- 分别请求算法
- HUD 显示：
  - 不鸣
  - 吃方案 A
  - 吃方案 B
  - 碰
  - 杠

并给出综合排序。

## 9. 最终判断

一句话总结：

- 这个算法足够支撑你的“推荐出牌”核心需求
- 对“吃碰杠建议”也能支撑，但前提是 `Crawler` 负责做局面模拟和分支比较

所以答案是：

- `能满足需求，但鸣牌建议不是直接喂一个字段就完事，而是要做一层分支求值`

如果下一步开始正式接入，推荐先做：

- `tile` 字符串到算法整数编码的转换
- `Crawler state -> 算法 request` 的组包函数
- `摸牌后调用算法并把前 3 项结果显示到 HUD`
